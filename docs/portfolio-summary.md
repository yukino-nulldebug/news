# Morning News ポートフォリオサマリー

## 何を評価してほしいか

外部APIに依存する小さなCLIバッチを、再現性、失敗時の継続、ログ、秘密情報保護、テストまで含めて、公開できる形に設計・実装した点を見てほしいです。

Morning News は、RSS/APIから国内外ニュースとマーケット情報を取得し、朝に確認しやすいMarkdownレポートとして保存するPython製CLIバッチアプリです。

## 作ったもの

- 国内外ニュース、マーケット情報、警告、注意事項を1つのMarkdownレポートにまとめる。
- `sample` モードではAPIキーなしで動作確認できる。
- `api` モードではRSS/APIから実データ取得を試せる。
- 外部取得の部分失敗、データ欠損、取得全滅をログ、警告、終了コードで切り分ける。

## 評価軸

| 評価してほしい力 | 実装で示したこと |
| --- | --- |
| 再現性 | `sample` / `api` を分け、APIキーがない第三者でも確認できる経路を用意した。 |
| 外部依存の扱い | timeout、retry、Provider分岐、マーケットAPIの連続取得間隔を設定化した。 |
| 失敗設計 | 欠損値は `N/A` と警告で継続し、外部データ全滅は終了コード `2` で明示した。 |
| 公開安全性 | `.env` をGit管理外にし、APIキー、トークン、認証ヘッダーをログ・レポートに出さない。 |
| 品質保証 | unit、integration、provider、securityのpytestとGitHub Actionsで回帰確認する。 |

## AI活用で示したいこと

このプロジェクトでは、AIを「コードを代わりに書かせる道具」ではなく、設計整理、実装案、レビュー、テスト観点の洗い出しを加速する開発パートナーとして使いました。

評価してほしい点は、AIを使ったこと自体ではなく、AIの出力をそのまま採用せず、仕様判断、安全判断、差分確認、テスト追加まで自分で行ったことです。

| 工程 | AIで加速したこと | 自分で判断したこと | 成果物 |
| --- | --- | --- | --- |
| Prompt | 課題、要件、受け入れ条件の整理 | APIキーなしで見せるため `sample` モードを標準にする | 要件定義、基本設計 |
| Build | 設定、Provider、レポート生成、pytestの実装案 | 小さな単位で差分を確認し、既存構造に合う形だけ採用する | `src/`、`tests/` |
| Review | コードレビュー、セキュリティ観点、仕様漏れの洗い出し | 外部データ全滅を成功扱いにしない仕様へ修正する | `main.py`、`test_api_mode.py` |
| Verify | テスト観点、README構成、見せ方の改善案 | 修正を回帰テストとドキュメントへ反映する | `pytest 85 tests`、README、HTML表紙 |

## AI協働で改善した具体例

- AIレビューで、`api` モードにおいて使える外部データがない場合でも成功扱いになり得る点を検出した。
- `current_value` がないマーケット情報は利用可能な外部データとして数えないように修正した。
- 同じ問題が戻らないよう、終了コード `2` を確認する結合テストを追加した。
- 古い `calculation_warning` が残る可能性も潰し、単体テストで固定した。

## 設計判断と証拠

| 課題 | 判断 | 実装・確認ポイント |
| --- | --- | --- |
| APIキーがない閲覧者が試せない | `sample` モードを標準確認経路にする | `src/config/settings.py`、`sample_data/`、`reports/sample-report.md` |
| 外部取得失敗を成功に見せてしまう | 部分失敗は警告、全滅は終了コード `2` | `main.py`、`tests/integration/test_api_mode.py` |
| 無料APIの制限に当たりやすい | 連続取得間隔を `.env` で調整可能にする | `MARKET_REQUEST_INTERVAL_SECONDS`、Alpha Vantage Providerテスト |
| 公開リポジトリで秘密情報が漏れる | URL、警告、ログ、レポートをマスクする | `src/utils/http_client.py`、`tests/security/test_secret_masking.py` |
| 仕様変更で挙動が戻る | 異常系・境界値をpytestで固定する | `tests/unit/`、`tests/integration/`、`.github/workflows/tests.yml` |

## 実物で確認できるもの

- `public/index.html`: 採用側が最初に見るためのポートフォリオ表紙。
- `morning-news/reports/sample-report.md`: 生成されるMarkdownレポートの例。
- `morning-news/README.md`: セットアップ、sample実行、api実行、テスト手順。
- `docs/requirements/要件定義.md`: 目的、MVPスコープ、受け入れ条件。
- `docs/basic-design/基本設計.md`: 実行モード、データ構造、モジュール責務、エラー方針。
- `docs/detail-design/`: Phaseごとの実装対象、入出力、テスト観点。
- `docs/learning/`: 問題、調査、実装、学びの形で残した学習ログ。

## レビューと改善の進め方

- 設計、実装、テスト、READMEを分けて整備し、第三者が追える形にした。
- レビューで見つかった仕様の穴を、コード修正だけでなく回帰テストまで追加して塞いだ。
- AIは設計整理やレビュー補助に使い、採用する仕様、安全性、公開方針は自分で判断した。
- 変更メモは `docs/learning/` に学習ログとしてまとめ、改善理由と学びまで追える形にした。
