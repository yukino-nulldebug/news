# AI協働で設計・実装を進めた記録

## 目的

AIを使いながら設計・実装したことを、採用向けに説明できる学習ログとして残す。

見せたいのは「AIに作らせたこと」ではなく、AIを使って設計整理、実装案、レビュー、テスト観点を早く出し、最終的に自分で判断して検証したこと。

## 発生した問題

AIを使った開発は、説明の仕方を間違えると次のように見えやすい。

- どこまで自分で考えたのか分からない。
- AIの出力をそのまま使っただけに見える。
- 仕様判断、セキュリティ判断、テスト確認が弱く見える。

そのため、AI活用の事実だけでなく、AIをどの工程で使い、何を自分で判断したかを成果物に残す必要があった。

## 調査

プロジェクト内の成果物を、AI活用の工程に対応づけて確認した。

| 工程 | AIで加速したこと | 自分で判断したこと | 成果物 |
| --- | --- | --- | --- |
| 要件整理 | 課題、機能、受け入れ条件の分解 | APIキーなしで見せるため `sample` モードを標準確認経路にする | `docs/requirements/要件定義.md` |
| 設計 | 実行モード、データ構造、例外方針の整理 | CLIバッチとMarkdown出力に絞る | `docs/basic-design/基本設計.md` |
| 実装 | Provider、設定、レポート生成、pytestの実装案 | 既存構造に合う形だけ採用し、差分を確認する | `morning-news/src/`、`morning-news/tests/` |
| レビュー | 仕様漏れ、セキュリティ、テスト不足の指摘 | 外部データ全滅を成功扱いにしない仕様へ修正する | `main.py`、`test_api_mode.py` |
| 検証 | テスト観点とREADME構成の提案 | 回帰テスト、README、HTML表紙へ反映する | `pytest`、`README.md`、`public/index.html` |

## 実装

AI協働を見せるため、プロジェクト内では次を整備した。

| 対象 | 実装・整備した内容 |
| --- | --- |
| `README.md` | このプロジェクトで示すことに、AI協働、失敗設計、公開安全性を明記した。 |
| `docs/portfolio-summary.md` | AIで加速したことと、自分で判断したことを表にした。 |
| `public/index.html` | 採用側が最初に見るHTMLに、AI-assisted development のセクションを置いた。 |
| `docs/learning/` | 相談内容を学習ログとして残し、問題から学びまで追えるようにした。 |
| `tests/security/` | AIレビューで出やすい秘密情報漏えいリスクを、テストで確認できる形にした。 |

開発フローとしては、次の形にした。

```text
Prompt
↓
Build
↓
Review
↓
Verify
```

各工程で、AIの出力をそのまま信じず、差分、テスト、公開安全性を確認してから採用した。

## 学んだこと

AI活用を強みにするには、「AIを使った」と書くだけでは足りない。

強く見えるのは、次のように検証可能な形で残している場合だった。

- AIで要件を分解した。
- 実装案を既存構造に合わせて取捨選択した。
- レビューで見つかった仕様の穴を修正した。
- 同じ問題が戻らないようにテストを追加した。
- 秘密情報をAIやログへ出さない方針を守った。
- READMEやHTMLで、第三者が確認しやすい導線を作った。

## 次の改善

- 主要なAI相談ごとに、採用した案・採用しなかった案・理由を短く残す。
- 実装前のプロンプトと、実装後の差分を対応づける。
- AIレビューで出た指摘を、`docs/changes/` または `docs/learning/` に追加する。
- テスト実行結果と修正理由を、学習ログに追記できるようにする。
