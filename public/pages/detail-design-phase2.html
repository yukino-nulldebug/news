4. ログ設計

出力先とフォールバック

  • 標準出力: 実行中に確認するためのログ。
  • logs/app.log: 実行履歴として残すログ。
  • logs/ が存在しない場合は作成する。
  • ファイル出力に失敗した場合、標準出力のみで処理を継続する。

ログ形式

YYYY-MM-DD HH:mm:ss LEVEL FEATURE_ID PROCESS_NAME MESSAGE
日時JST
LEVELINFO / WARNING / ERROR
FEATURE_IDF-01 から F-10
PROCESSmodule.function
MESSAGE件数、失敗理由、保存先

    process["各処理"] --> info["INFO: 起動 / 件数 / 保存"]

5. 例外設計

Phase 2 では、失敗箇所と機能IDをログに残しやすくするため、標準例外をアプリケーション専用例外へ変換する。 

classDiagram
    class MorningNewsError {
      message
      feature_id
      process_name
      recoverable
    }
    MorningNewsError <|-- ConfigError
    MorningNewsError <|-- DataLoadError
    MorningNewsError <|-- DataValidationError
    MorningNewsError <|-- SummaryFormatError
    MorningNewsError <|-- MarketCalculationError
    MorningNewsError <|-- ReportGenerationError
    MorningNewsError <|-- ReportWriteError

データ・設定
  • ConfigError: 設定読み込み・設定値不正
  • DataLoadError: ファイル読み込み失敗、JSON解析失敗
  • DataValidationError: 必須項目欠損、型不正
整形・計算
  • SummaryFormatError: 概要文整形失敗
  • MarketCalculationError: 前日比・変化率計算失敗
  • 継続可能な場合は警告として扱う。
生成・保存・ログ
  • ReportGenerationError: Markdown生成失敗
  • ReportWriteError: レポート保存失敗
  • LoggingSetupError: 原則送出せず標準出力へフォールバック

7. 関数設計

main.py

  • main(): 全体処理を実行し、終了コードを返す。
  • build_report_data(settings, logger, result): データ取得、整形、計算、警告集計を行う。
  • log_exception(): 例外情報をログと実行結果に記録する。
  • log_execution_summary(): 実行結果サマリーを出力する。

utils/

  • setup_logger(log_dir): 標準出力と logs/app.log のロガーを作成する。
  • log_info() / log_warning() / log_error(): 機能ID付きログを出す。
  • record_warning() / record_error(): 集計を更新する。

news/

  • load_news_items(file_path, feature_id): 指定JSONを読み込み、機能IDで失敗箇所を識別する。
  • fetch_sample_news(settings): 実装する場合は薄いラッパーに限定する。
  • format_news_items(items, max_length): 概要文なし件数を集計可能にする。

market/

  • calculate_change(current_value, previous_close): 型不正は例外、previous_close == 0 は None を返す。
  • calculate_market_changes(items): 計算不可件数と対象指標名を返せるようにする。

report/

  • generate_report(report_data): 必須キー欠損時は専用例外を送出する。
  • generate_market_comments(items): 生成件数をログ対象として扱う。
  • write_report(markdown, report_path): OSError を保存用例外へ変換する。

基本呼び出し

domestic_news = load_news_items(settings.news_jp_path, "F-01")
global_news = load_news_items(settings.news_global_path, "F-02")

9. エラー処理詳細

0

レポート保存まで完了した。警告があっても保存できた場合は正常終了として扱う。

1

設定、Markdown生成、レポート保存など、出力処理に失敗した。

2

必須データの読み込みまたは検証に失敗し、レポートを作成できなかった。

発生箇所条件動作ログ終了コード
ログ初期化logs/app.log に書き込めない標準出力のみで継続WARNING F-090
ニュース読み込みJSONが存在しない、必須項目欠損処理停止ERROR F-01/F-022
概要文整形summary が空代替文を設定して継続WARNING F-030
概要文整形max_length <= 0処理停止ERROR F-031
マーケット計算previous_close == 0N/A として継続WARNING F-050
レポート保存ファイル書き込み失敗処理停止ERROR F-081

10. ReportData への反映

ReportData

  • generated_atレポート生成日時
  • modePhase 2 では sample
  • news_domestic国内ニュース一覧
  • news_global海外ニュース一覧
  • marketsマーケット情報一覧
  • warningsレポートに表示する継続可能な警告
  • execution_summaryログと確認用の実行結果概要

表示してよい警告

  • 概要文が空だったため代替文を設定した。
  • 変化率を計算できなかったため N/A と表示した。
  • 一部の任意項目が欠損していたため代替値を設定した。

ログのみに残す情報

  • ファイルシステムの詳細なエラーメッセージ。
  • 内部パスの長いトレース。
  • APIキー、トークン、認証情報。
  • サンプルJSONの本文全体。

13. 確認観点 / 14. 受け入れ条件

確認観点

  • python main.py が終了コード 0 で終了する。
  • logs/app.log に起動、読み込み、整形、計算、生成、保存、終了のログがある。
  • 警告発生時に WARNING ログと警告件数が残る。
  • データ不備で ERROR と終了コード 2 になる。
  • 生成または保存失敗で終了コード 1 になる。
  • APIキー相当の値がログに含まれない。

受け入れ条件

  • logs/app.log が作成または追記される。
  • 日時、ログレベル、機能ID、処理名、メッセージが出力される。
  • 正常終了時に読み込み件数、保存先、実行結果サマリーが残る。
  • 異常終了時に失敗した機能ID、処理名、失敗理由が残る。
  • 終了コード 1 と 2 を区別する。
  • ロガー初期化失敗時も標準出力で継続できる。
ログ作成logs/app.log が作成または追記される。
形式日時、レベル、機能ID、処理名、メッセージがある。
正常系件数、保存先、サマリーが残る。
警告系継続しつつ警告件数を集計する。
