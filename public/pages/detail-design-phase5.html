1. 詳細設計の目的

Phase 5 は、Phase 4 まで手動確認していた主要処理を pytest で自動化する。 外部ネットワーク、実APIキー、実 .env に依存しないテストを優先する。

02

2. Phase 5 の対象範囲

Phase 5 で実装する

  • requirements-dev.txt と pytest 実行環境
  • tests/ 配下の単体、Provider、結合、securityテスト
  • tmp_path / monkeypatch を使うfixture
  • sampleモードとモックapiモードの結合テスト
  • JSON欠損、数値不正、保存失敗などの異常系
  • APIキーや認証URLを出力しない回帰テスト

Phase 5 では実装しない

  • 実APIへの自動E2Eテスト
  • 定時実行テスト
  • Web UI、通知、AI要約のテスト
  • 厳密なカバレッジ閾値運用
  • 外部サービス利用規約の自動検証
  • NewsAPI の実Provider化
Phase 5-A

sampleモード中心

formatter、calculator、generator、writer、execution result、sample結合テストを先に整備する。

Phase 5-B

Provider / api / security

settings、fetcher、HTTP、Provider、api結合、セキュリティ回帰テストを追加する。

03

3. テスト基本方針

テストは仕様ベースで確認する。出力ファイルは一時ディレクトリへ閉じ込め、既存の reports/ や logs/ を壊さない。

Unit

概要文整形、変化率計算、Markdown生成、保存先生成、実行結果集計を検証する。

Provider

RSS、Alpha Vantage、HTTP共通処理を monkeypatch で固定応答に差し替える。

Integration

APP_MODE=sample と、HTTPをモックした APP_MODE=api を確認する。

Security

ログ、警告、レポート、例外メッセージに秘密値が出ないことを確認する。

flowchart TD
    tests["python3 -m pytest"] --> unit["tests/unit"]
    tests --> providers["tests/providers"]
    tests --> integration["tests/integration"]
    tests --> security["tests/security"]
    unit --> tmp["tmp_path"]
    providers --> patch["monkeypatch"]
    integration --> noNetwork["No external network"]
    security --> mask["masked secrets"]

07

7. Provider・HTTPテスト

Providerのテストは外部サービスに接続しない。取得関数を差し替えて、正規化と警告の仕様を確認する。

RSS Provider

正常RSS、HTML概要、日時欠損、title欠損、重複URL、HTTP失敗、limitを検証する。

NewsAPI Provider

Phase 5では未対応スタブのみ。通信せず、空配列と警告を返すことに限定する。

Alpha Vantage

APIキー未設定、正常Quote、API制限Note、部分失敗、キー値マスク、連続取得間の待機を検証する。

HTTP Client

sanitize_url()、200応答、JSON解析失敗、404、Timeout、リトライを確認する。

マスク対象

apikey、api_key、token、access_token を *** に置換する。

後続任意

NewsAPIの実Provider化、GDELT、GNews、実API任意E2EはPhase 5の外に置く。

08

8. 結合テスト設計

APP_MODE=sample

  • main.main() が終了コード 0 を返す。
  • tmp_path/reports/YYYY-MM-DD.md を作る。
  • 国内ニュース、海外ニュース、マーケット情報、市況コメント、注意事項を含む。
  • 再実行しても同一ファイルが上書きされ、成功する。
Integration

APP_MODE=api

  • RSSとマーケットAPI成功時は保存まで完了する。
  • RSSのみ成功、マーケットAPIキー未設定でも警告付きで成功する。
  • NewsAPI provider指定では通信せず未対応警告を返す。
  • 外部データ全件0件なら保存せず終了コード 2。
flowchart TD
    start["main.main"] --> mode{"APP_MODE"}
    mode -->|sample| sample["sample_data"]
    mode -->|api| api["mocked external data"]
    sample --> report["report saved"]
    api --> any{"1件以上取得?"}
    any -->|Yes| report
    any -->|No| exit2["exit code 2 / no report"]
    report --> exit0["exit code 0"]

09

9. 異常系とセキュリティ回帰

JSON異常

ファイルなし、壊れたJSON、items 型不正、必須項目欠損。

設定異常

SUMMARY_MAX_LENGTH=0、NEWS_LIMIT=abc、不正RSS URL。

保存異常

レポート保存先が書き込めない場合は ReportWriteError。

全取得失敗

apiモードで外部データ全件0件なら終了コード 2。

APIキー

test-secret-key を出さない。

認証クエリ

apikey=*** にする。

token

token=*** にする。

.env

全文や実値を出力しない。

Authorization

ヘッダー内容をログに出さない。

10

10. テストデータ設計

正常系は既存の sample_data/ を使う。境界値と異常系は Phase 5 で追加する sample_data_edge/ に分ける。

sample_data/

APIキーなしで動かす正常系sampleモードの確認に使う。

sample_data_edge/

欠損、空概要、数値不正などの境界値・異常系テストに使う。

tmp_path

ファイル欠損、JSON不正、保存失敗など、実ファイルを壊すケースに使う。

sample_data_edge/ は通常実行では使用しない。
pytest実行時のみ参照し、公開後の利用者が sample モードを確認するときは sample_data/ を使う。

11. 実装順序

Phase 5-A でsampleモード中心の回帰テストを固め、Phase 5-B でProvider、apiモード、セキュリティ回帰を足す。

5-ADev deps

requirements-dev.txt と conftest.py。

5-ACore unit

formatter、calculator、generator、writer、execution result。

5-Asample

sampleモード結合テストと手動実行回帰。

5-BProvider

settings、fetcher、HTTP、RSS、Alpha Vantage、NewsAPIスタブ。

5-BSecurity

api結合、全取得失敗、秘密情報非出力。

12

12. 受け入れ条件

pytest成功

python3 -m pytest が終了コード 0 で完了する。

外部依存なし

外部ネットワーク、実APIキー、実 .env に依存しない。

主要処理

概要文整形、変化率計算、Markdown生成、市況コメント生成を単体テストで確認できる。

Provider

RSS、Alpha Vantage、NewsAPI未対応スタブ、HTTP共通処理を確認できる。

結合

sample保存、api成功、api全取得失敗時の終了コード 2 を確認できる。

異常系

JSON欠損、数値不正、設定異常、保存失敗を検証している。

機密情報

APIキー、token、未マスク認証URLがログ・レポート・警告に出ない。
